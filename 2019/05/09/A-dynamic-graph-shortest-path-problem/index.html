A-dynamic-graph-shortest-path-problem

发表于 2019-05-09 |

一个简单的算法问题讨论

Introduction & Definition

在最近的三次OO中介绍了一个动态图问题，定义如下:

  1. 节点，一个节点，用编号表示
  2. 边，两个节点之间边是一个二元组$ < fr,to > $ 两个节点之间相连，用$to(e)$、$fr(e)$表示二元组的两个元素。
  3. 线路，线路是一个有标号边序列$ \{ p_i \} $，记标号为$pid$ 其中，序列中任意相邻两个$p_i$和$p _ {i+1}$ 满足 $to ( p_i ) = fr( p _ {i+1} ) $，线路上的每条边赋予与线路相同标号。
  4. 边的线路号，记做$pid(e)$，表示在哪些线路中出现，是一个 集合 。
  5. 路径是一个序列$ \{ e_i \} $ 其中，路径中任意相邻两个$e_i$和$e _ {i+1}$ 满足 $to ( e_i ) = fr( e _ {i+1} ) $
  6. 路径的边权，路径的边权等于每条边的权和，记做$ ev ( \{ e_i \} )$
  7. 路径的换乘权，对于一个路径$ \{ p_i \}$ ， 设$ C = \{ i | pid(p_i) \cap pid(p _ {i+1} ) = \emptyset \} $, 集合$C$的大小$|C|$是路径的换乘权，记做$ cv ( \{ p_i \} ) $

那么现在的问题是，对一张图支持

  1. 动态的加入一个线路
  2. 动态的删除一个线路
  3. 查询任意两点之间路径的边权和的最小值
  4. 查询任意两点之间路径的换乘权+边权的最小值

Solution

问题的关键在于处理换乘。 可以将每条线路上的同标号节点，视作不同的节点。 做拆点。

一个朴素的想法是，对所有的相同标号的点（属于不同线路的点）之间，两两连边，权值为换乘边的边权。做最短路。

这样的坏处是:

加的边的数量是平方级别的

Dijkstra

这是一个通用的SSSP算法，复杂度为$O( N + M \log N)$ 其中$N$表示点的个数，$M$表示边的个数。可以看到，这样的算法适合 稀疏图。 故加的边太多，对Dijkstra算法是有害的。

此外，SSSP算法是通用的，应当写成抽象的模板，一个言简意赅的写法如下，此处省略了变量的定义若干，对象的构造若干，给出了保证复杂度正确的最短路写法。

此处有贴实现code的必要，工程问题要考虑实现细节，只谈设计不落地是不现实的。

Trick 1

首先，为了改变边的数量过多，一个合理的想法是，对于每个点设立换乘站，其含义和真实的地铁情况类似，两条公共站的地铁，实际上换乘是要先进入本站的换乘平台再离开本站换乘平台。

用图示如下

graph LR
    A --- |One| D
    B --- |Two| D
    C --- |Three| D 
    D --- |One| E
    D --- |Two| F
    D --- |Three| G

拆点后

graph LR
 D0 --> A
 D0 --> B
 D0 --> C
 D0 --> E
 D0 --> F
 D0 --> G
 A --- |One| D1
 B ---|Two| D2
 C ---|Three| D3 
 D1 ---|One| E
 D2 ---|Two| F 
 D3 ---|Three| G 
 D1 --> D0
 D2 --> D0
 D3 --> D0

注意到拆点后的图是换乘站和其他点的关系是有向的，即一条原始边$fr \leftrightarrow to$
要拆成$fr \rightarrow fr’$,$to \rightarrow to’$,$fr \leftrightarrow to$, $ fr’ \rightarrow to $, $fr \leftarrow to’$ 多出四条边。

这样我们的换乘边的数量就是平方级别的

Trick 2

使用时间戳缓存

因为有多种边权和换乘权的查询，而每次查询都做一次单源最短路是得不偿失的。故可以考虑缓存这些数据。

利用$timestamp_i$表示以$i$为源点的最短路数据缓存的时间戳，利用$tm _ {count}$表示当前修改次数，记当前询问类型为（$id$）

每次查询前判断

  1. 如果$ (timestamp_i \gg 4) == tm _ {count} $ 表示当前缓存的时间戳有效
    • 如果$ (timestamp_i \& (1 \ll id)) == (1 \ll id) $ 表示当前缓存了此类询问

根据情况更新$ timestamp_i $

Trick 3

注意到$fr \rightarrow to$ 和$ to \rightarrow fr$的查询是本质一样的，这样能减少很多缓存的必要。

Experience

在实验中，我的平均时间是$1.5 second$。
